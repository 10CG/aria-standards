# 编辑原则与规范

> **文档类型**: 方法论规范
> **版本**: 1.0.0
> **最后更新**: 2025-12-28

## 概述

本文档定义了设计系统文档编辑的基本原则、禁止事项和推荐做法，旨在确保设计系统的一致性、可维护性和团队协作效率。

## 核心原则

### 1. 单一事实来源 (SSOT)

**定义**: 每个设计令牌只在核心设计系统中定义一次，其他文档通过引用使用。

**理由**: 避免不一致、简化维护、确保变更可追溯。

### 2. 层次化架构

**定义**: 遵循三层文档架构，每层有明确的职责和依赖关系。

**理由**: 职责分离、降低耦合、提高可扩展性。

### 3. 引用而非复制

**定义**: 使用令牌引用（如 `primary.green`）而非硬编码值（如 `#24D298`）。

**理由**: 保持同步、便于全局更新、减少错误。

## 严禁事项

### ❌ 禁止 1: 重复定义设计令牌

**禁止行为**：
- 在平台应用规范中重新定义色彩值、字体大小等基础令牌
- 在技术实现文档中硬编码设计参数
- 创建与核心设计系统冲突的新令牌定义

**错误示例**：
```markdown
# mobile-ui-spec.md (平台应用规范)
❌ 主品牌色：#24D298（玉青色）
```

**正确做法**：
```markdown
# mobile-ui-spec.md (平台应用规范)
✅ 主品牌色：`primary.green` (Day: 玉青色)
   - 应用场景：主要按钮、品牌标识
   - 交互状态：Hover 时透明度 80%
```

**影响**: 重复定义导致不一致、维护困难、变更风险高。

---

### ❌ 禁止 2: 跨层级直接引用

**禁止行为**：
- 技术实现文档直接引用核心设计系统的值，绕过平台应用规范
- 第一层引用第二层或第三层的内容（逆向依赖）

**错误示例**：
```css
/* web-implementation.md (技术实现) */
❌ .button-primary {
  background-color: #24D298; /* 直接硬编码核心令牌值 */
}
```

**正确做法**：
```css
/* web-implementation.md (技术实现) */
✅ .button-primary {
  background-color: var(--primary-green); /* 引用 CSS 变量 */
}

/* CSS 变量定义部分 */
:root {
  --primary-green: #24D298; /* 来自: primary.green (Day) */
}
```

**影响**: 跨层引用破坏架构分层，难以追溯令牌来源。

---

### ❌ 禁止 3: 无授权修改核心系统

**禁止行为**：
- 未经设计系统负责人审核直接修改核心设计系统
- 单方面创建与现有规范冲突的新规范
- 绕过变更流程进行紧急修改

**正确做法**：
1. 提交设计系统变更请求
2. 与相关平台负责人沟通影响范围
3. 通过设计系统负责人审核
4. 按照变更工作流执行

**影响**: 未授权修改可能导致多平台不一致、破坏现有实现。

---

### ❌ 禁止 4: 创建冲突规范

**禁止行为**：
- 在不同平台创建相互冲突的设计规范
- 定义与核心设计系统不一致的令牌解释
- 为相同语义的令牌使用不同的名称

**错误示例**：
```markdown
# web-ui-spec.md
❌ 主品牌色：`primary.blue` (#1E88E5)  <!-- 与移动端的 primary.green 冲突 -->
```

**正确做法**：
```markdown
# web-ui-spec.md
✅ 主品牌色：`primary.green`
   - Web 端应用方式：用于顶部导航栏、主要 CTA 按钮
   - 与移动端保持一致
```

**影响**: 冲突规范破坏跨平台一致性，损害用户体验。

## 推荐做法

### ✅ 推荐 1: 建立清晰的引用链

**做法**：每个文档明确标注其依赖的上级文档，使用相对链接建立引用关系。

**示例**：
```markdown
# mobile-ui-spec.md

> **设计令牌来源**: [核心设计系统](../core-design-system.md)
> **本文档层次**: 第二层 - 平台应用规范
> **下游文档**: [移动端技术实现](../implementation/mobile-implementation.md)

## 色彩映射

### 主品牌色
- **令牌**: `primary.green`
- **定义**: [core-design-system.md#色彩系统](../core-design-system.md#色彩系统)
- **应用**: 主要按钮、品牌标识、重点强调元素
```

**优势**: 清晰的引用链便于追溯、维护和理解。

---

### ✅ 推荐 2: 使用一致的命名约定

**做法**：采用统一的令牌命名规范，如 `category.subcategory.property`。

**示例**：
```markdown
# 核心设计系统

## 色彩令牌
- `primary.green`: 主品牌色
- `primary.blue`: 辅助品牌色
- `background.primary`: 主背景色
- `background.secondary`: 次背景色
- `text.primary`: 主文本色
- `text.secondary`: 次文本色

## 间距令牌
- `spacing.xs`: 4px
- `spacing.sm`: 8px
- `spacing.md`: 16px
- `spacing.lg`: 24px
- `spacing.xl`: 32px
```

**优势**: 一致的命名提高可读性，减少混淆。

---

### ✅ 推荐 3: 在文档开头说明位置

**做法**：在每个设计文档的开头明确说明其在三层架构中的位置。

**示例**：
```markdown
# Web 端技术实现指南

> **文档类型**: 技术实现文档（第三层）
> **依赖文档**: [Web 端应用规范](../platforms/web-ui-spec.md)
> **目标受众**: 前端开发工程师
> **技术栈**: React + TypeScript + styled-components
```

**优势**: 明确的定位帮助读者理解文档角色和使用方式。

---

### ✅ 推荐 4: 提供变更影响分析

**做法**：在提交设计系统变更时，分析并说明对下游文档和实现的影响。

**示例**：
```markdown
# 变更请求：更新主品牌色

## 变更内容
- `primary.green` (Day): #24D298 → #1FD190

## 影响范围
- **第二层文档**:
  - mobile-ui-spec.md: 需更新色彩应用示例
  - web-ui-spec.md: 需更新色彩应用示例

- **第三层实现**:
  - mobile-implementation.md: 需更新 Dart 常量定义
  - web-implementation.md: 需更新 CSS 变量值

- **代码影响**:
  - 前端: 重新编译 CSS 变量
  - 移动端: 重新构建颜色常量文件

## 迁移计划
1. Week 1: 更新核心设计系统
2. Week 2: 更新平台应用规范
3. Week 3: 更新技术实现文档和代码
4. Week 4: 全面测试和部署
```

**优势**: 全面的影响分析确保平稳过渡，减少意外问题。

## 编辑前检查清单

### 编辑核心设计系统前

- [ ] 确认变更的必要性和业务价值
- [ ] 评估对所有平台的影响范围
- [ ] 与相关平台负责人沟通
- [ ] 检查是否会破坏现有平台实现
- [ ] 准备变更通知和迁移指南
- [ ] 获得设计系统负责人审核批准

### 编辑平台应用规范前

- [ ] 检查核心设计系统是否已有所需令牌
- [ ] 如需新令牌，先向核心设计系统申请
- [ ] 确认不与其他平台规范冲突
- [ ] 使用令牌引用而非硬编码值
- [ ] 添加适当的设计系统文档链接
- [ ] 说明平台特定的应用场景

### 编辑技术实现文档前

- [ ] 确认平台应用规范支持该实现
- [ ] 检查技术栈的兼容性和可行性
- [ ] 使用变量/常量引用设计令牌
- [ ] 提供清晰的代码示例
- [ ] 考虑向后兼容性和迁移路径
- [ ] 添加必要的注释说明令牌来源

## 常见错误与修正

### 错误 1: 硬编码颜色值

**错误**：
```jsx
// ❌ 错误
const Button = styled.button`
  background-color: #24D298;
`;
```

**修正**：
```jsx
// ✅ 正确
const Button = styled.button`
  background-color: var(--primary-green);
`;
```

---

### 错误 2: 在平台规范中重新定义令牌

**错误**：
```markdown
# mobile-ui-spec.md
❌ ## 色彩定义
- 主品牌色: #24D298
```

**修正**：
```markdown
# mobile-ui-spec.md
✅ ## 色彩映射
- 主品牌色: `primary.green` (来自核心设计系统)
  - 应用场景: 主要按钮、品牌标识
```

---

### 错误 3: 缺少令牌来源说明

**错误**：
```css
/* ❌ 错误：不知道这个值从哪来 */
:root {
  --primary-color: #24D298;
}
```

**修正**：
```css
/* ✅ 正确：明确令牌来源 */
:root {
  /* 来自: core-design-system.md -> primary.green (Day) */
  --primary-green: #24D298;
}
```

## 团队协作原则

1. **沟通优先**: 重大变更前与相关方充分沟通
2. **文档先行**: 先更新文档再修改代码
3. **版本同步**: 保持文档版本与代码版本一致
4. **审核机制**: 所有变更需经过指定审核人批准
5. **变更通知**: 及时通知受影响的团队成员
