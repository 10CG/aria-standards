# 三层文档架构规范

> **文档类型**: 方法论规范
> **版本**: 1.0.0
> **最后更新**: 2025-12-28

## 概述

设计系统采用三层文档架构，每层有明确的职责范围和依赖关系。这种分层确保了设计令牌的单一事实来源，同时允许不同平台根据需要进行适配。

## 架构图

```
┌─────────────────────────────────────────┐
│  第一层：核心设计系统                    │
│  - 设计令牌定义（颜色、字体、间距等）    │
│  - 平台无关的抽象规范                   │
│  - 所有其他文档的引用源                 │
└─────────────────┬───────────────────────┘
                  │ 引用
                  ↓
┌─────────────────────────────────────────┐
│  第二层：平台应用规范                    │
│  - 引用第一层设计令牌                   │
│  - 平台特定的应用指南                   │
│  - 交互规范和适配规则                   │
└─────────────────┬───────────────────────┘
                  │ 引用
                  ↓
┌─────────────────────────────────────────┐
│  第三层：技术实现文档                    │
│  - 将设计令牌转换为代码                 │
│  - 具体的实现指南和示例                 │
│  - 平台特定的技术细节                   │
└─────────────────────────────────────────┘
```

## 第一层：核心设计系统

### 职责

**主要职责**：
- 定义所有基础设计令牌（Design Tokens）
- 色彩系统、字体系统、间距系统、组件规范
- 提供平台无关的抽象定义
- 作为所有其他设计文档的单一事实来源

**不应包含**：
- 平台特定的实现细节
- 具体的代码示例
- 技术栈相关的内容

### 典型内容结构

```markdown
# 核心设计系统

## 色彩系统
- primary.green: #00E5A0 (Night) / #24D298 (Day)
- primary.blue: #3B82F6
- ...

## 字体系统
- font.family.primary: "Inter", sans-serif
- font.size.h1: 32px
- ...

## 间距系统
- spacing.xs: 4px
- spacing.sm: 8px
- ...
```

### 编辑权限

| 角色 | 权限 |
|------|------|
| 首席设计师/设计系统负责人 | 主要负责 (R) |
| UX 设计师、品牌设计师 | 协作参与 (C) |
| 技术负责人、产品负责人 | 审核指导 (A) |

## 第二层：平台应用规范

### 职责

**主要职责**：
- 引用第一层的设计令牌
- 说明如何在特定平台（Web/Mobile/Desktop）应用设计令牌
- 定义平台特有的交互规范
- 提供具体界面设计指南

**不应包含**：
- 重新定义设计令牌的基础值
- 具体的代码实现
- 与核心设计系统冲突的规范

### 典型内容结构

```markdown
# [平台名称] 应用规范

> **设计令牌来源**: [核心设计系统](../core-design-system.md)

## 色彩映射
- 主品牌色：使用 `primary.green`
  - 应用场景：主要按钮、品牌标识
  - 平台适配：支持深色/浅色模式自动切换

## 平台特定规范
- Mobile 特有：底部导航栏设计
- Web 特有：侧边栏导航设计
```

### 编辑权限

| 角色 | 权限 |
|------|------|
| 对应平台的 UX 设计师 | 主要负责 (R) |
| 前端/移动端开发工程师、产品经理 | 协作参与 (C) |
| 设计系统负责人 | 审核指导 (A) |

## 第三层：技术实现文档

### 职责

**主要职责**：
- 将设计令牌转换为具体代码
- 提供技术栈特定的实现指南
- 包含代码示例和最佳实践
- 说明如何在实际项目中应用设计系统

**不应包含**：
- 硬编码的设计值
- 绕过设计令牌的直接实现
- 与上层规范冲突的实现

### 典型内容结构

```markdown
# [平台] 技术实现指南

> **应用规范来源**: [平台应用规范](../platform-spec.md)

## CSS 变量定义
\`\`\`css
:root {
  --primary-green: #24D298; /* 引用: primary.green (Day) */
  --spacing-sm: 8px;        /* 引用: spacing.sm */
}
\`\`\`

## React 组件示例
\`\`\`jsx
const Button = styled.button\`
  background-color: var(--primary-green);
  padding: var(--spacing-sm);
\`;
\`\`\`
```

### 编辑权限

| 角色 | 权限 |
|------|------|
| 前端架构师/移动端架构师 | 主要负责 (R) |
| 开发工程师、设计师 | 协作参与 (C) |
| 技术负责人 | 审核指导 (A) |

## 跨层依赖规则

### ✅ 正确的依赖关系

```
第一层定义: primary.green = #24D298
    ↓ 引用
第二层应用: 主品牌色使用 `primary.green`
    ↓ 引用
第三层实现: --primary-green: var(--primary-green)
```

### ❌ 禁止的依赖关系

- **跳层引用**：第三层直接引用第一层，绕过第二层
- **逆向依赖**：第一层引用第二层或第三层的内容
- **重复定义**：在第二层或第三层重新定义设计令牌的基础值

## 版本控制策略

### 变更传播顺序

1. **第一层变更** → 通知所有平台负责人 → 更新第二层 → 更新第三层
2. **第二层变更** → 检查是否需要新令牌 → 如需要则先更新第一层 → 更新第三层
3. **第三层变更** → 检查第二层是否支持 → 如不支持则先更新第二层

### 版本号管理

- 第一层变更：主版本号变更（1.0.0 → 2.0.0）
- 第二层变更：次版本号变更（1.0.0 → 1.1.0）
- 第三层变更：修订版本号变更（1.0.0 → 1.0.1）

## 实施建议

1. **从第一层开始**：先建立核心设计系统，再逐步扩展
2. **保持简洁**：每层只包含该层应有的内容
3. **明确引用**：在文档中明确标注引用来源
4. **定期审查**：确保三层之间的一致性
