## 🔄 1. 文档同步标准机制

### 1.1 同步触发条件

```yaml
自动同步触发:
  条件1_任务完成:
    触发时机: Step 7进度更新时
    同步范围:
      - UPM文档（必需）
      - 任务文档状态更新（必需）
      - 架构文档（如有变更）

  条件2_阶段转换:
    触发时机: Stage状态变更时
    同步范围:
      - UPM的UPMv2-STATE片段
      - 项目周期文档（milestone-summary.md）
      - 质量验证报告（quality-review.md）

  条件3_质量指标变化:
    触发时机: KPI显著变化（≥5%变化）
    同步范围:
      - UPM的kpiSnapshot字段
      - quality-review.md

  条件4_风险识别:
    触发时机: 识别到新风险或风险等级变化
    同步范围:
      - UPM的risks字段
      - 项目周期文档的risks章节

手动同步触发:
  场景1_计划调整:
    操作: 人工修改计划文档
    同步要求: 必须回写UPM的nextCycle字段

  场景2_架构重构:
    操作: 重大架构变更
    同步要求:
      - 更新架构文档
      - 记录ADR
      - 更新UPM的lastUpdateRef
```

### 1.2 同步内容标准

```yaml
UPM文档同步内容:
  必需同步字段:
    - stage: 阶段状态
    - cycleNumber: 循环编号
    - lastUpdateAt: 更新时间戳
    - lastUpdateRef: 引用指针
    - stateToken: 状态哈希（重新计算）
    - kpiSnapshot: 质量指标快照

  建议同步字段:
    - nextCycle: 下一循环候选任务
    - risks: 风险列表
    - pointers.lifecycle.current: 当前周期文档路径

  可选同步字段:
    - 模块特定字段（由扩展标准定义）

项目周期文档同步内容:
  progress-report.md:
    - 完成任务列表
    - KPI指标详细数据
    - 遇到的问题和解决方案
    - 下周期计划调整

  quality-review.md:
    - 测试覆盖率详细报告
    - 代码质量检查结果
    - 性能指标验证数据
    - 技术债务记录

  milestone-summary.md:
    - 里程碑目标回顾
    - 实际达成情况对比
    - 经验教训总结
    - 后续计划调整

架构文档同步内容:
  ARCHITECTURE.md:
    - 架构变更记录
    - 组件依赖关系图更新
    - 设计决策更新

  ADR文档:
    - 新增决策记录
    - 决策状态更新（proposed→accepted→deprecated）
```

### 1.3 同步时机规范

```yaml
同步时机决策树:
  场景: AI完成开发任务
    ├─ 技术验证通过？
    │   ├─ 是 → 进入架构同步（Step 5）
    │   └─ 否 → 修复问题，重新验证
    │
    ├─ 架构文档需要更新？
    │   ├─ 是 → 更新ARCHITECTURE.md + ADR（如需）
    │   └─ 否 → 跳过架构同步
    │
    ├─ Git提交（Step 6）
    │   └─ 使用标准提交格式
    │
    └─ 进度更新（Step 7）
        ├─ 第一步：写入项目周期文档
        │   - 路径: docs/project-lifecycle/week{N}/
        │   - 文件: progress-report.md或其他
        │
        ├─ 第二步：回写UPM的UPMv2-STATE
        │   - 校验stateToken
        │   - 更新必需字段
        │   - 重新计算stateToken
        │
        └─ 第三步：验证同步完整性
            - 检查所有引用有效
            - 验证文档一致性

最佳实践时机:
  ✅ 推荐: 任务完成后立即同步
    - 信息最新鲜，记忆最准确
    - 避免信息遗漏

  ⚠️ 可接受: 小任务批量完成后同步
    - 适用于多个小任务快速完成
    - 建议不超过3个任务

  ❌ 禁止: 跨循环延迟同步
    - 信息易丢失
    - 难以追溯
```

### 1.4 自动化同步流程

```yaml
自动化同步脚本（概念设计）:
  触发: post-commit hook或CI/CD pipeline

  流程:
    Step 1: 检测变更范围
      - git diff分析修改文件
      - 识别涉及的模块
      - 确定需要同步的文档

    Step 2: 提取同步信息
      - 从commit message提取关键信息
      - 解析技术验证结果
      - 收集质量指标

    Step 3: 更新UPM文档
      - 读取当前stateToken
      - 校验token有效性
      - 更新字段并重新计算token
      - 写入UPM

    Step 4: 验证同步结果
      - 检查引用完整性
      - 验证格式正确性
      - 生成同步报告

同步失败处理:
  - 记录失败原因
  - 创建待处理任务
  - 通知相关人员
  - 保留原始状态以便回滚
```

## 📊 2. 智能任务调度机制

### 2.1 任务优先级动态调整

```yaml
优先级评估框架:
  四个维度:
    业务价值（40%权重）:
      高价值: 核心功能、用户痛点解决
      中价值: 功能增强、体验优化
      低价值: 次要功能、内部工具

    技术风险（30%权重）:
      高风险: 架构重构、核心模块变更
      中风险: 新技术引入、复杂算法
      低风险: 常规开发、配置调整

    依赖关系（20%权重）:
      阻塞性: 被多个任务依赖
      前置性: 依赖其他任务但不阻塞他人
      独立性: 无明显依赖关系

    资源可用性（10%权重）:
      立即可用: 无需等待资源
      部分可用: 需要协调资源
      不可用: 资源瓶颈

优先级计算公式:
  Priority = (业务价值 × 0.4) + (技术风险反向 × 0.3) + (依赖性 × 0.2) + (资源可用性 × 0.1)

  其中:
    - 技术风险反向 = 1 - 技术风险归一化值
    - 依赖性：阻塞性=1.0, 前置性=0.5, 独立性=0.3
    - 各维度归一化为0-1之间

动态调整触发条件:
  条件1_质量下降:
    触发: 测试覆盖率<85% 或 构建失败
    调整: 技术债务修复任务优先级↑↑

  条件2_风险升级:
    触发: 风险等级从P2/P3升至P0/P1
    调整: 风险缓解任务优先级↑↑

  条件3_依赖解除:
    触发: 依赖任务完成
    调整: 被依赖任务优先级↑

  条件4_资源变化:
    触发: 新资源可用或资源减少
    调整: 相关任务优先级↑或↓
```

### 2.2 下一循环候选集获取规则

```yaml
获取策略（多级优先）:
  Level 1 - 轻量级索引（首选）:
    来源: UPM的nextCycle.candidates字段
    优点:
      - 读取速度快（<1秒）
      - 已经预筛选过
      - 包含候选理由

    格式:
      nextCycle:
        intent: "下一循环的目标意图"
        candidates:
          - id: "mobile-B6-2#task-analytics-chart"
            rationale: "高优先级数据可视化功能"
          - id: "mobile-B7#ai-offline-online-switch"
            rationale: "关键AI功能切换，用户需求强烈"
        constraints:
          - "依赖B5/B6已验证"
          - "内存溢出问题已缓解"

    使用条件:
      - candidates非空
      - 候选任务状态为pending
      - 依赖关系已满足

  Level 2 - 任务文档扫描（备选）:
    来源: {module}/project-planning/detailed-development-tasks.md
    触发: nextCycle.candidates为空或不满足条件

    扫描策略:
      过滤条件:
        - 状态: pending或ready
        - 优先级窗口: P0-P2（不包括P3除非无其他任务）
        - 依赖检查: 前置依赖已完成

      排序规则:
        1. P0优先级任务（紧急且重要）
        2. 依赖关系（阻塞性任务优先）
        3. 业务价值（核心功能优先）
        4. 技术风险（低风险优先，除非高风险任务阻塞）

      结果限制: 返回前5-10个候选任务

  Level 3 - 全局任务搜索（极少使用）:
    来源: 所有模块的任务文档
    触发: Level 2也无满足条件的任务

    场景:
      - 跨模块依赖解除
      - 资源临时释放
      - 计划外紧急任务

    注意: 此级别扫描成本高，应尽量避免

任务元数据要求:
  必需字段:
    - id: 唯一标识符（格式：{module}-{group}#{task-id}）
    - status: pending/in_progress/completed/blocked
    - priority: P0/P1/P2/P3
    - dependencies: 依赖任务ID列表

  建议字段:
    - businessValue: 业务价值评分（1-10）
    - technicalRisk: 技术风险评分（1-10）
    - estimatedEffort: 预估工作量（小时）
    - requiredSkills: 所需技能标签

优化建议:
  ✅ 推荐: 维护nextCycle.candidates
    - 在每次进度更新（Step 7）时更新
    - 基于当前状态和优先级动态调整
    - 提供候选理由，便于AI决策

  ⚠️ 注意: 避免过度扫描
    - 不要每次都全量扫描任务文档
    - 使用pointers.tasks快速定位
    - 利用git log分析最近活跃任务
```

### 2.3 任务依赖管理

```yaml
依赖类型定义:
  强依赖（must-have）:
    说明: 必须等待前置任务完成
    阻塞: 是
    示例: "UI组件依赖API端点实现"

  弱依赖（nice-to-have）:
    说明: 建议等待但可并行
    阻塞: 否
    示例: "详细文档依赖功能实现，但可先写框架"

  互斥依赖（exclusive）:
    说明: 不能同时进行
    阻塞: 是
    示例: "数据库迁移期间不能修改schema"

依赖声明格式:
  tasks:
    - id: "mobile-B6-2#analytics-chart"
      dependencies:
        - taskId: "backend-A3#analytics-api"
          type: "must-have"
          reason: "需要API提供数据"
        - taskId: "mobile-B6-1#data-model"
          type: "must-have"
          reason: "需要数据模型定义"
        - taskId: "mobile-B5#ui-components"
          type: "nice-to-have"
          reason: "可复用组件库，但可以自己实现"

依赖检查流程:
  Step 1: 获取任务依赖列表
  Step 2: 查询每个依赖任务的状态
  Step 3: 判断阻塞性依赖是否满足
  Step 4: 如果所有must-have依赖已完成 → 任务可执行
  Step 5: 如果有阻塞 → 标记为blocked状态
```

### 2.4 调度算法

```yaml
贪心调度算法（推荐用于日常开发）:
  目标: 最大化每个循环的产出价值

  步骤:
    1. 获取候选任务集（Level 1或Level 2）
    2. 过滤blocked任务
    3. 按优先级排序
    4. 选择Top N任务（N=循环容量）
    5. 验证资源可行性
    6. 返回最终任务列表

  优点: 简单高效，适合大部分场景
  缺点: 可能忽略长期规划

关键路径法（用于里程碑规划）:
  目标: 确保里程碑按时完成

  步骤:
    1. 识别里程碑目标
    2. 反向推导关键路径
    3. 优先调度关键路径任务
    4. 并行调度非关键路径任务
    5. 监控进度偏差，动态调整

  优点: 保证里程碑按时完成
  缺点: 灵活性较低

混合调度策略（推荐用于复杂项目）:
  规则:
    - 80%容量: 使用贪心算法选择高优先级任务
    - 20%容量: 保留给技术债务、重构、探索性任务
    - 关键里程碑前2-3个循环: 切换为关键路径法

  优点: 平衡短期产出和长期健康
```

## 🎯 3. 智能混合策略集成

### 3.1 核心原则

```yaml
原则1_主体一致性原则:
  定义: 确保AI准确理解项目的真实状态

  要求:
    - UPM文档必须反映项目实际状态
    - 不能存在过时或错误的关键信息
    - AI读取的信息必须是可信的

  示例:
    ❌ 错误: UPM显示"Phase 3 - Development"，但实际已进入Testing
    ✅ 正确: 及时更新UPM的stage字段为"Phase 3 - Testing"

原则2_历史透明性原则:
  定义: 保留原计划，便于学习和复盘

  要求:
    - 原始计划不得直接删除或覆盖
    - 使用明确标记区分"原计划"与"实际执行"
    - 修正时必须注明修正原因和时间

  示例:
    ✅ 正确做法:
      ## 原计划（2025-12-01制定）
      - 任务A: 开发用户认证模块（预计3天）
      - 任务B: 集成第三方支付（预计2天）

      ## 实际执行（2025-12-05更新）
      - 任务A: ✅ 已完成（实际用时4天，因发现安全漏洞需额外加固）
      - 任务B: 🔄 进行中（依赖任务A延迟，预计2025-12-07完成）

      **偏差原因**: 任务A发现安全漏洞，需要额外1天修复
      **调整措施**: 任务B顺延1天，不影响整体里程碑

原则3_决策可追溯原则:
  定义: 记录每次调整的原因，便于未来决策参考

  要求:
    - 每次计划调整必须记录原因
    - 决策上下文要完整（为什么这样调整？考虑了哪些因素？）
    - 结果验证（调整是否有效？有何经验教训？）

  示例:
    ### 决策记录 #001
    **时间**: 2025-12-05 15:30
    **决策**: 将任务B从Cycle 7推迟到Cycle 8
    **原因**:
      - 依赖任务A延迟1天完成
      - Cycle 7剩余时间不足以完成任务B
      - 任务B非关键路径，不影响里程碑
    **预期影响**:
      - Cycle 7产出减少1个功能
      - Cycle 8负载增加，需注意容量
    **实际验证**:
      - （待Cycle 8完成后填写）
```

### 3.2 偏差识别机制

```yaml
偏差类型定义:
  时间偏差:
    定义: 实际完成时间与计划时间不符
    阈值:
      - 轻微: ±20%以内
      - 中等: ±20%-50%
      - 严重: >±50%

    示例:
      计划3天，实际4天 → 中等偏差（+33%）
      计划2天，实际5天 → 严重偏差（+150%）

  范围偏差:
    定义: 实际实现范围与计划范围不符
    类型:
      - 范围缩减: 计划功能未实现
      - 范围扩展: 实现了计划外功能
      - 范围变更: 替代方案实现

    示例:
      计划: 实现A+B功能
      实际: 仅实现A功能 → 范围缩减
      实际: 实现A+B+C功能 → 范围扩展

  质量偏差:
    定义: 实际质量指标与目标不符
    维度:
      - 测试覆盖率偏差
      - 性能指标偏差
      - 代码质量偏差

    阈值:
      - 可接受: 目标的90%-110%
      - 需改进: 目标的80%-90%或110%-120%
      - 不可接受: <80%或>120%

自动检测触发:
  检测点1_任务完成时:
    检查项:
      - 对比计划工作量 vs 实际工作量
      - 对比计划范围 vs 实际范围
      - 检查质量指标是否达标

    触发: 任何偏差≥中等级别

  检测点2_Cycle结束时:
    检查项:
      - 汇总本Cycle所有任务偏差
      - 分析偏差趋势（偏差是否增加？）
      - 评估对下一Cycle的影响

    触发: 偏差率>30%（即30%以上任务有偏差）

  检测点3_里程碑检查时:
    检查项:
      - 里程碑目标达成率
      - 关键路径任务偏差
      - 质量门禁通过情况

    触发: 任何关键指标未达标

AI偏差感知接口:
  查询接口:
    功能: AI查询当前是否存在偏差
    路径: UPM文档 → "偏差管理"章节

    返回信息:
      - 偏差列表
      - 偏差等级
      - 建议措施

  报告接口:
    功能: AI报告新识别的偏差
    格式:
      ```yaml
      deviation:
        type: "time"  # time/scope/quality
        severity: "moderate"  # minor/moderate/severe
        task: "mobile-B6-2#analytics-chart"
        planned: "3 days"
        actual: "4 days"
        reason: "发现数据格式兼容性问题，需额外处理"
      ```
```

### 3.3 文档修正策略

```yaml
修正策略矩阵:
  轻微偏差（<20%）:
    修正方式: 记录偏差，不修改原计划
    文档更新:
      - 在实际执行章节注明偏差
      - 原计划保持不变

    示例:
      ## 原计划
      - 任务A: 3天

      ## 实际执行
      - 任务A: ✅ 3.5天完成
      **注**: 实际用时略超计划，因代码审查发现小问题需修复

  中等偏差（20%-50%）:
    修正方式: 追溯修正计划，保留原计划历史
    文档更新:
      - 创建"计划修正记录"章节
      - 更新当前计划为修正后版本
      - 原计划移至"历史计划"章节

    示例:
      ## 当前计划（修正版，2025-12-05更新）
      - 任务A: 4天（原计划3天）
      - 任务B: 3天（原计划2天）

      ## 历史计划（2025-12-01初版）
      - 任务A: 3天
      - 任务B: 2天

      ## 修正原因
      - 任务A: 发现安全漏洞需额外加固
      - 任务B: 依赖任务A延迟，工作量评估偏低

  严重偏差（>50%）:
    修正方式: 触发重新规划
    文档更新:
      - 创建"重规划决策记录"
      - 全面评估剩余任务
      - 调整里程碑和Cycle计划

    触发流程:
      1. 暂停当前计划执行
      2. 召集团队（或AI深度分析）
      3. 重新评估剩余工作量
      4. 调整计划并记录决策
      5. 更新UPM和项目文档

追溯修正标准:
  修正时机:
    ✅ 应该修正:
      - 发现原计划有明显错误
      - 实际偏差严重影响后续计划
      - 为了帮助AI做出更准确的决策

    ❌ 不应修正:
      - 仅为了"看起来更准确"
      - 轻微偏差且不影响后续
      - 偏差是正常波动范围

  修正边界:
    - 只修正未来的计划，不修改已完成任务的记录
    - 保留原计划的完整副本
    - 明确标注修正版本和时间

历史保留标准:
  永久保留:
    - 里程碑计划
    - 重大决策记录
    - Phase级别的计划

  周期保留（至少保留6个月）:
    - Cycle级别的计划
    - 任务级别的详细记录
    - 偏差分析和修正记录

  可归档（保留至项目结束）:
    - 日常进度记录
    - 详细开发日志

版本管理机制:
  版本号规则:
    格式: v{MAJOR}.{MINOR}

    MAJOR变更:
      - Phase转换
      - 里程碑重新规划
      - 重大架构调整

    MINOR变更:
      - Cycle计划调整
      - 中等偏差修正
      - 任务优先级调整

  示例:
    - v1.0: Cycle 1初始计划
    - v1.1: Cycle 1中期调整（任务B推迟）
    - v2.0: Cycle 2开始（Phase 3进入Development）
```

### 3.4 实施流程

```yaml
阶段1_偏差识别:
  触发点: 任务完成时、Cycle结束时、里程碑检查时

  执行步骤:
    1. AI或人工对比计划 vs 实际
    2. 计算偏差百分比
    3. 分类偏差类型（时间/范围/质量）
    4. 评估偏差等级（轻微/中等/严重）
    5. 分析偏差原因

  输出:
    - 偏差识别报告
    - 偏差等级判定
    - 原因分析

阶段2_修正执行:
  触发点: 识别到中等或严重偏差

  决策树:
    IF 偏差等级 = 轻微 THEN
      记录偏差，不修正计划
    ELSE IF 偏差等级 = 中等 THEN
      追溯修正计划，保留历史
    ELSE IF 偏差等级 = 严重 THEN
      触发重新规划流程
    END IF

  修正操作:
    1. 备份原计划
    2. 更新计划文档
    3. 记录修正原因和决策
    4. 更新UPM的nextCycle字段（如影响下一循环）
    5. 通知相关团队成员

阶段3_质量验证:
  验证点: 修正完成后

  检查清单:
    □ 原计划已备份到"历史计划"章节
    □ 修正原因已明确记录
    □ 修正后的计划逻辑合理
    □ 相关文档已同步更新（UPM、任务文档）
    □ 版本号已更新

  验证通过标准:
    - 所有检查项通过
    - 修正决策有充分依据
    - 不影响文档的主体一致性

阶段4_持续维护:
  维护周期: 每个Cycle结束时

  维护任务:
    1. 回顾本Cycle的偏差情况
    2. 分析偏差趋势（是否增加？重复原因？）
    3. 总结经验教训
    4. 改进估算方法
    5. 更新下一Cycle的计划质量

  持续改进:
    - 建立偏差模式库（哪些类型任务容易偏差？）
    - 优化工作量估算（基于历史数据）
    - 提升计划准确性
```

## 🔍 4. 偏差管理与文档修正机制

### 4.1 偏差分类与等级

```yaml
时间偏差分类:
  提前完成:
    等级判定:
      - 轻微: 提前10%-20%
      - 中等: 提前20%-40%
      - 显著: 提前>40%

    可能原因:
      - 工作量高估
      - 技术储备充分
      - 复用现有方案

    处理建议:
      - 轻微: 正常记录，调整估算参考
      - 中等: 分析原因，优化估算方法
      - 显著: 检查是否范围缩减，验证质量

  延迟完成:
    等级判定:
      - 轻微: 延迟10%-20%
      - 中等: 延迟20%-50%
      - 严重: 延迟>50%

    可能原因:
      - 工作量低估
      - 技术难题
      - 依赖阻塞
      - 范围蔓延

    处理建议:
      - 轻微: 记录偏差，调整后续估算
      - 中等: 分析原因，修正计划，调整后续任务
      - 严重: 重新评估，可能需要重新规划

范围偏差分类:
  范围缩减:
    等级判定:
      - 轻微: 非核心功能未实现（<20%）
      - 中等: 部分核心功能未实现（20%-50%）
      - 严重: 大部分功能未实现（>50%）

    原因分析:
      - 时间不足被迫缩减
      - 技术难题无法攻克
      - 需求变更取消部分功能

    处理:
      - 轻微: 记录缩减内容，评估影响
      - 中等: 修正计划，未实现功能推迟到下一Cycle
      - 严重: 触发重新规划，重新评估里程碑

  范围扩展:
    等级判定:
      - 轻微: 小幅增加（<20%）
      - 中等: 中等增加（20%-50%）
      - 显著: 大幅增加（>50%）

    原因分析:
      - 发现新需求
      - 技术优化带来额外功能
      - 范围蔓延（需警惕）

    处理:
      - 轻微: 记录增加内容，评估价值
      - 中等: 分析必要性，调整下一Cycle容量
      - 显著: 检查是否范围蔓延，必要时拒绝

质量偏差分类:
  质量低于目标:
    维度:
      - 测试覆盖率不达标
      - 性能指标不达标
      - 代码质量问题多

    等级判定:
      - 轻微: 90%-100%达标
      - 中等: 80%-90%达标
      - 严重: <80%达标

    处理:
      - 轻微: 技术债务记录，下一Cycle修复
      - 中等: 立即分配修复任务，不能推迟
      - 严重: 阻止任务完成，必须立即修复

  质量高于目标:
    说明: 通常是好事，但需分析成本

    检查:
      - 是否过度设计？
      - 是否影响交付速度？
      - 是否值得投入？

    处理:
      - 记录高质量实践
      - 评估成本收益
      - 适度即可，避免过度优化
```

### 4.2 修正决策树

```yaml
决策树入口:
  输入: 偏差类型、偏差等级、偏差原因
  输出: 修正策略、具体操作步骤

决策流程:
  START
  │
  ├─ Q1: 偏差等级？
  │   ├─ 轻微 → 策略A: 记录偏差，不修正计划
  │   ├─ 中等 → Q2
  │   └─ 严重 → 策略C: 触发重新规划
  │
  ├─ Q2: 是否影响后续任务？
  │   ├─ 不影响 → 策略A: 记录偏差
  │   └─ 影响 → Q3
  │
  ├─ Q3: 影响范围？
  │   ├─ 仅本Cycle → 策略B1: 追溯修正本Cycle计划
  │   ├─ 下一Cycle → 策略B2: 追溯修正并调整nextCycle
  │   └─ 里程碑 → 策略C: 触发重新规划
  │
  END

策略A_记录偏差不修正:
  适用场景:
    - 轻微偏差
    - 不影响后续任务
    - 偏差在可接受范围

  操作步骤:
    1. 在"实际执行"章节注明偏差
    2. 分析偏差原因
    3. 更新偏差统计
    4. 保持原计划不变

  文档模板:
    ```markdown
    ## 实际执行
    - 任务A: ✅ 完成（3.5天，原计划3天）
      **偏差**: +0.5天（+17%）
      **原因**: 代码审查发现小问题需修复
      **影响**: 无，不影响后续任务
    ```

策略B1_追溯修正本Cycle:
  适用场景:
    - 中等偏差
    - 影响本Cycle后续任务
    - 不影响下一Cycle

  操作步骤:
    1. 备份原计划到"历史计划"章节
    2. 更新"当前计划"为修正版
    3. 记录修正原因和决策
    4. 调整本Cycle剩余任务安排
    5. 更新UPM的cycleNumber相关信息
    6. 增加版本号MINOR

  文档模板:
    ```markdown
    ## 当前计划（v1.1修正版，2025-12-05）
    - 任务A: ✅ 已完成（4天）
    - 任务B: 🔄 进行中（3天，原2天）
    - 任务C: ⏳ 待开始（推迟到下一Cycle）

    ## 历史计划（v1.0初版，2025-12-01）
    - 任务A: 3天
    - 任务B: 2天
    - 任务C: 2天

    ## 修正决策 #001
    **时间**: 2025-12-05
    **原因**: 任务A延迟1天，导致本Cycle时间不足
    **决策**: 任务C推迟到下一Cycle，任务B调整为3天
    **依据**: 任务B可完成核心功能，任务C非关键路径
    ```

策略B2_追溯修正并调整nextCycle:
  适用场景:
    - 中等偏差
    - 影响下一Cycle
    - 不影响里程碑

  操作步骤:
    1. 执行策略B1的所有步骤
    2. 更新UPM的nextCycle.candidates
    3. 调整下一Cycle的计划容量
    4. 记录对下一Cycle的影响

  文档模板:
    ```markdown
    [策略B1的内容]

    ## 对下一Cycle的影响
    - 任务C推迟至Cycle 8
    - Cycle 8负载增加1个任务
    - 建议：Cycle 8减少1个低优先级任务以平衡负载

    UPM更新:
      nextCycle:
        intent: "完成任务C + 原计划任务"
        candidates:
          - id: "mobile-B6-3#task-C"
            rationale: "从Cycle 7推迟，高优先级"
          - id: "mobile-B7-1#task-D"
            rationale: "原计划任务"
        constraints:
          - "注意容量，必要时调整任务D"
    ```

策略C_触发重新规划:
  适用场景:
    - 严重偏差
    - 影响里程碑
    - 需要全面重新评估

  操作步骤:
    1. 暂停当前计划执行
    2. 创建"重规划决策记录"文档
    3. 全面评估剩余工作量
    4. 重新制定Cycle计划
    5. 必要时调整里程碑
    6. 更新所有相关文档
    7. 增加版本号MAJOR

  文档模板:
    ```markdown

    ## 触发原因
    **时间**: 2025-12-05
    **偏差情况**: Cycle 7严重延迟，完成率仅60%

    - 任务D: 3天
    - 任务E: 4天
    - 总计: 12天，原计划剩余7天

    ### 里程碑调整
    - Milestone 3: 推迟1周（从2025-12-15到2025-12-22）
    - 原因: 工作量低估 + 技术难题

    ## 新计划（v2.0）
    - Cycle 7: 完成任务C（5天）
    - Cycle 8: 完成任务D+E（7天）
    - Cycle 9: 缓冲与测试（3天）

    ## 经验教训
    - 新技术栈工作量估算偏低
    - 需要增加技术预研时间
    - 建议：未来类似任务增加50%缓冲
    ```

### A.2 智能混合策略FAQ

**Q1: 什么时候应该修正计划？**

A: 当偏差等级为中等或严重，且影响后续任务或里程碑时，应该修正计划。轻微偏差仅记录即可。

**Q2: 如何保证不破坏历史透明性？**

A: 始终保留原计划的完整副本，使用版本化管理，明确标注修正时间和原因。

**Q3: AI如何感知偏差？**

A: AI通过读取UPM文档的"偏差管理"章节或项目周期文档的偏差记录来感知偏差。

**Q4: 偏差率多高算正常？**

A: 轻微偏差率<30%为正常，30%-50%需关注，>50%需重点改进。

**Q5: 如何避免范围蔓延？**

A: 严格遵循计划，新需求需经过评审，明确区分"必须"和"可选"功能。
