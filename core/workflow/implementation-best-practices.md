## 🤖 1. AI操作最佳实践

### 1.1 高效查询策略

```yaml
快速查询模式（日常开发）:
  目标: 5秒内获取关键信息，开始任务

  查询步骤:
    Step 1: 读取UPM的UPMv2-STATE片段
      路径: {module}/project-planning/unified-progress-management.md
      读取范围: YAML frontmatter（前200行）
      提取信息:
        - module: 当前模块
        - stage: 当前阶段
        - cycleNumber: 当前循环
        - kpiSnapshot: 质量指标快照
        - nextCycle.candidates: 候选任务列表

    Step 2: 快速扫描当前项目状态章节
      位置: UPM文档的"当前项目状态"章节
      提取信息:
        - 活跃任务列表
        - 近期完成任务
        - 当前风险

    Step 3: 决策是否需要详细分析
      IF 任务明确且无风险 THEN
        开始执行
      ELSE IF 任务复杂或有风险 THEN
        切换为详细分析模式
      END IF

  适用场景:
    - 熟悉的常规任务
    - 无明显风险的小任务
    - 快速迭代开发

  示例查询:
    "读取mobile UPM的UPMv2-STATE，识别下一个P0/P1任务"

详细分析模式（复杂任务）:
  目标: 全面理解项目状态，做出准确决策

  查询步骤:
    Step 1: 完整读取UPM文档
      重点章节:
        - UPMv2-STATE
        - 当前项目状态
        - 最近循环摘要
        - 风险与问题
        - 技术债务

    Step 2: 读取相关计划文档
      IF 需要理解战略规划 THEN
        读取 mo-production-development-plan.md
      IF 需要任务详细信息 THEN
        读取 detailed-development-tasks.md

    Step 3: 读取架构文档
      IF 任务涉及架构变更 THEN
        读取 ARCHITECTURE.md + 相关ADR

    Step 4: 分析依赖关系
      检查:
        - 前置依赖是否完成
        - 是否会阻塞其他任务
        - 资源是否可用

    Step 5: 风险评估
      评估:
        - 技术风险
        - 时间风险
        - 质量风险

  适用场景:
    - 复杂的新功能
    - 架构重构任务
    - 高风险任务
    - 第一次接触项目

应急查询模式（问题处理）:
  目标: 快速定位问题，找到解决线索

  查询步骤:
    Step 1: 识别问题领域
      问题类型:
        - 构建失败 → 读取最近的提交和技术验证记录
        - 测试失败 → 读取质量验证文档
        - 性能问题 → 读取性能指标历史
        - 文档不一致 → 读取最近的进度更新记录

    Step 2: 定位相关文档
      使用UPM的pointers快速定位:
        - pointers.architecture: 架构文档
        - pointers.tasks: 任务详细文档
        - pointers.lifecycle.current: 当前周期文档

    Step 3: 分析问题历史
      查看:
        - 类似问题的解决记录
        - 技术债务清单
        - 风险缓解措施

  适用场景:
    - 紧急问题修复
    - 质量回归
    - 部署失败

偏差感知查询（智能混合策略）:
  目标: 识别计划与实际的偏差

  查询步骤:
    Step 1: 读取UPM的偏差管理章节
      检查:
        - 是否有未解决的偏差记录
        - 偏差等级和影响

    Step 2: 对比计划 vs 实际
      对比维度:
        - 时间: 计划工作量 vs 实际工作量
        - 范围: 计划功能 vs 实际功能
        - 质量: 目标指标 vs 实际指标

    Step 3: 评估偏差影响
      判断:
        - 是否需要修正计划
        - 是否影响后续任务
        - 是否需要上报

  适用场景:
    - Cycle结束时
    - 里程碑检查时
    - 任务完成后验证
```

### 1.2 状态更新最佳实践

```yaml
及时更新原则:
  规则: 任务完成后立即更新，不延迟

  理由:
    - 信息最新鲜，记忆最准确
    - 避免信息遗漏或混淆
    - 保持文档与代码同步

  实施:
    - 完成任务后，立即执行Step 7进度更新
    - 不等待多个任务批量完成
    - 特殊情况（如小任务）可批量，但不超过3个

  示例:
    ✅ 正确: 完成任务A → 立即更新UPM → 开始任务B
    ❌ 错误: 完成任务A → 完成任务B → 完成任务C → 才更新UPM

准确更新原则:
  规则: 更新信息必须准确反映实际情况

  关键字段准确性:
    - stage: 必须与实际阶段一致
    - cycleNumber: 按规则递增，不能跳跃
    - lastUpdateAt: 使用准确的ISO 8601时间戳
    - kpiSnapshot: 基于实际验证结果，不能猜测

  验证方法:
    - 技术验证必须实际运行，不能假设
    - 测试覆盖率必须实际测量，不能估计
    - 性能指标必须实际profiling，不能臆测

  示例:
    ✅ 正确: 运行flutter test --coverage，实测覆盖率87.2%
    ❌ 错误: 猜测覆盖率大概80%左右

完整更新原则:
  规则: 更新必须完整，不能遗漏关键信息

  必需更新字段（UPMv2-STATE）:
    - ✅ lastUpdateAt
    - ✅ lastUpdateRef
    - ✅ stateToken（重新计算）
    - ✅ kpiSnapshot

  建议更新字段:
    - ✅ nextCycle.candidates（为下一循环准备）
    - ✅ risks（如有新风险或风险变化）

  周期文档完整性:
    - 写入详细的进度报告
    - 记录遇到的问题和解决方案
    - 更新质量验证结果

  示例:
    ✅ 完整更新:
      1. 写入progress-report.md（详细叙事）
      2. 回写UPM的UPMv2-STATE（机读接口）
      3. 更新任务文档状态
      4. 验证同步完整性

    ❌ 不完整更新:
      仅更新UPM，不写周期文档

偏差记录原则:
  规则: 发现偏差必须记录，不能隐藏

  记录时机:
    - 任务完成时发现偏差
    - Cycle结束时汇总偏差
    - 里程碑检查时分析偏差

  记录内容:
    - 偏差类型和等级
    - 偏差原因分析
    - 采取的修正措施
    - 经验教训总结

  诚实原则:
    - 不粉饰偏差
    - 不隐瞒失误
    - 实事求是记录

  示例:
    ✅ 正确:
      "任务B延迟1天，原因是低估了数据迁移复杂度，
      下次类似任务需增加50%缓冲"

    ❌ 错误:
      "任务B按时完成"（实际延迟但不记录）
```

### 1.3 文档生成最佳实践

```yaml
项目周期文档生成:
  强制性检查（使用检查清单）:
    □ 确定当前周数/循环编号
      方法: 从UPM的cycleNumber字段获取

    □ 检查week{N}目录是否存在
      路径: docs/project-lifecycle/week{N}/
      不存在则创建: mkdir -p docs/project-lifecycle/week{N}

    □ 使用标准命名（无week前缀）
      正确: plan.md, progress-report.md, quality-review.md
      错误: week7-plan.md

    □ 更新project-lifecycle/index.md索引
      添加新周期的条目和链接

    □ 在UPM中更新指针
      更新lifecycle.current指向新目录

  生成内容质量:
    进度报告（progress-report.md）:
      必需内容:
        - 完成任务列表（详细）
        - KPI指标详细数据
        - 遇到的问题和解决方案
        - 偏差记录和分析
        - 下周期计划调整

      格式规范:
        - 使用markdown标题层级
        - 任务列表使用checkbox格式
        - 数据使用表格呈现

    质量验证（quality-review.md）:
      必需内容:
        - 测试覆盖率详细报告
        - 代码质量检查结果
        - 性能指标验证数据
        - 技术债务记录

      数据来源:
        - 实际运行技术验证命令
        - 不得估计或猜测

  禁止行为:
    ❌ 在docs根目录或其他位置创建周文档
    ❌ 使用非标准命名格式
    ❌ 跳过索引更新步骤
    ❌ 重复创建同一周期的文档

架构文档更新:
  识别架构变更:
    何时更新ARCHITECTURE.md:
      - 新增模块或组件
      - 修改模块依赖关系
      - 技术栈变化
      - 重大设计模式变更

    何时创建ADR:
      - 有多个技术方案可选
      - 决策影响较大或长期
      - 需要记录决策上下文
      - 未来可能需要重新评估

  更新内容质量:
    架构图:
      - 保持最新，反映实际结构
      - 使用统一的图例和风格
      - 标注关键依赖关系

    组件说明:
      - 明确组件职责
      - 列出关键接口
      - 说明技术选型

    ADR记录:
      - 完整记录决策上下文
      - 列出评估的所有方案
      - 说明选择理由
      - 记录预期影响

提交消息生成:
  使用标准格式:
    格式: <type>(<scope>): <subject>

    type:
      - feat: 新功能
      - fix: bug修复
      - refactor: 重构
      - docs: 文档
      - test: 测试
      - chore: 构建/工具

    scope: 模块名或功能域

    subject: 简洁描述（<50字符）

  增强标记:
    [AI-DDD] 标记AI-DDD流程中的提交
    [Step-X] 标记七步流程的哪一步
    [Module-X] 标记模块

  示例:
    feat(mobile): 添加用户认证UI组件

    - 实现登录表单组件
    - 集成Flutter表单验证
    - 添加Widget测试覆盖

    [AI-DDD][Step-6][Mobile]
    Task: mobile-B6-1#auth-ui
    Quality: coverage=89%, build=green, lint=0
```

### 1.4 TDD Enforcer 最佳实践

```yaml
RED-GREEN-REFACTOR 循环:
  强制执行原则:
    - 测试必须先于业务代码编写
    - 测试失败前禁止编写实现代码
    - 测试通过后立即停止编码
    - 重构时必须保持测试通过

  Phase 执行规范:
    RED Phase (测试编写):
      目标: 编写会失败但应该通过的测试
      验证: 运行测试确认失败 (非语法错误)
      停止: 确认失败后立即停止，不编写业务代码

    GREEN Phase (最小实现):
      目标: 编写最少代码使测试通过
      验证: 运行测试确认通过
      停止: 通过后立即停止，不添加额外功能
      禁止: 优化代码、添加未测试功能

    REFACTOR Phase (代码重构):
      目标: 优化代码结构，保持功能不变
      验证: 每次重构后运行测试
      保持: 测试必须始终通过
      禁止: 修改功能需求

  常见错误:
    ❌ 在 GREEN Phase 添加"顺便"的功能
    ❌ 跳过 RED Phase 直接编写代码
    ❌ 重构时不运行测试
    ❌ 测试通过后继续添加未测试代码

  正确示例:
    ✅ RED:
       ```python
       # 编写测试
       def test_user_login_empty_username():
           with pytest.raises(ValueError):
               UserAuth.login("", "password")

       # 运行确认失败 (函数不存在)
       # 停止编码
       ```

    ✅ GREEN:
       ```python
       # 最小实现
       class UserAuth:
           @staticmethod
           def login(username, password):
               if not username:
                   raise ValueError("Username required")
               return True

       # 运行确认通过
       # 停止编码
       ```

    ✅ REFACTOR:
       ```python
       # 提取验证器
       class UsernameValidator:
           def validate(self, username):
               if not username:
                   raise ValueError("Username required")

       # 运行确认测试仍然通过
       ```

  语言支持:
    Python: pytest, unittest
    JavaScript/TypeScript: jest, mocha
    Dart (Flutter): flutter test
    Java: JUnit
    Go: testing package
```

### 1.5 Hooks 系统使用指南

```yaml
Hooks 配置位置:
  文件: aria/hooks/hooks.json
  结构:
    session-start: 会话开始时执行
    pre-commit: 提交前执行
    task-complete: 任务完成时执行
    subagent-stop: 子代理停止时执行

  使用场景:
    SessionStart Hook:
      目的: 确保开发环境就绪
      检查项:
        - Git 状态 (分支、未提交变更)
        - 依赖完整性
        - 环境变量配置
      示例配置:
        ```json
        {
          "session-start": {
            "enabled": true,
            "checks": ["git-status", "dependencies"],
            "on_fail": "warn"
          }
        }
        ```

    PreCommit Hook:
      目的: 确保提交质量
      检查项:
        - TDD 状态验证 (测试覆盖率)
        - 代码格式检查
        - 提交消息格式验证
      示例配置:
        ```json
        {
          "pre-commit": {
            "enabled": true,
            "checks": ["tdd-status", "lint", "commit-format"],
            "on_fail": "block"
          }
        }
        ```

    TaskComplete Hook:
      目的: 确保任务完成质量
      检查项:
        - 测试覆盖率 ≥ 85%
        - 文档同步状态
        - 架构文档更新确认
      示例配置:
        ```json
        {
          "task-complete": {
            "enabled": true,
            "checks": ["coverage", "docs-sync"],
            "coverage_threshold": 85
          }
        }
        ```

  最佳实践:
    ✅ 渐进式启用: 从 warn 模式开始，逐步过渡到 block
    ✅ 快速执行: Hook 应在 5 秒内完成
    ✅ 清晰输出: 失败时提供明确的修复建议
    ❌ 避免重复: 不检查已通过 CI 的内容
```

### 1.7 错误避免清单

```yaml
常见错误类型:
  错误1_读取文档不完整:
    表现: 仅读取部分章节，导致信息缺失
    后果: 决策错误，遗漏关键约束
    避免方法:
      - 使用标准查询流程
      - 至少读取UPMv2-STATE + 当前项目状态
      - 复杂任务使用详细分析模式

  错误2_状态更新不及时:
    表现: 任务完成后延迟更新或不更新
    后果: 文档过时，下一次AI查询获得错误信息
    避免方法:
      - 严格遵守"及时更新原则"
      - 完成任务立即执行Step 7

  错误3_stateToken未更新:
    表现: 修改UPM内容但忘记重新计算stateToken
    后果: 并发冲突检测失效，可能丢失数据
    避免方法:
      - 使用自动化工具计算stateToken
      - 每次UPM更新都检查stateToken

  错误4_周期文档路径错误:
    表现: 将周期文档放在非标准路径
    后果: AI无法找到，索引混乱
    避免方法:
      - 严格使用docs/project-lifecycle/week{N}/
      - 使用AI文档生成检查清单

  错误5_技术验证造假:
    表现: 未实际运行验证命令，猜测结果
    后果: 质量数据不可信，隐藏问题
    避免方法:
      - 必须实际运行验证命令
      - 记录验证命令输出
      - 定期审计质量数据

  错误6_偏差隐瞒:
    表现: 发现偏差但不记录，试图隐藏
    后果: 失去学习机会，重复犯错
    避免方法:
      - 实事求是记录偏差
      - 偏差不是失败，是学习机会
      - 分析偏差原因，改进估算

  错误7_依赖检查遗漏:
    表现: 开始任务前未检查依赖是否完成
    后果: 任务阻塞，浪费时间
    避免方法:
      - 使用开发前检查流程
      - 验证所有must-have依赖

  错误8_版本号混乱:
    表现: 版本号不按规则更新或重复
    后果: 无法追踪文档演进
    避免方法:

    □ 运行技术验证
    □ 写入周期文档
    □ 回写UPM状态
    □ 验证同步完整性

  Cycle结束时:
    □ 汇总偏差记录
    □ 分析偏差趋势
    □ 更新质量验证报告
    □ 规划下一Cycle

## 🚨 3. 故障处理与应急机制

### 3.1 常见问题快速解决

```yaml
问题1_文档状态不一致:
  症状:
    - UPM显示任务已完成，但任务文档显示pending
    - 质量指标不匹配
    - 引用链接失效

  诊断方法:
    1. 对比UPM vs 任务文档 vs 周期文档
    2. 检查最后更新时间
    3. 查看Git提交历史

  解决方案:
    Step 1: 确定哪个文档是最新准确的
      - 通常以UPM的stateToken和lastUpdateAt为准

    Step 2: 同步其他文档
      - 更新不一致的文档
      - 修正状态和数据

    Step 3: 验证一致性
      - 再次对比所有文档
      - 确保完全一致

    Step 4: 记录问题和修复
      - 记录问题原因
      - 改进流程避免重复

  预防措施:
    - 严格遵守同步顺序（先周期文档，后UPM）
    - 使用验证脚本自动检查一致性

问题2_AI查询结果异常:
  症状:
    - AI返回"找不到文档"
    - AI读取的信息明显过时
    - AI决策不符合预期

  诊断方法:
    1. 检查文档路径是否正确
    2. 验证UPM的pointers字段
    3. 查看文档最后更新时间

  解决方案:
    情况A_路径错误:
      - 检查docs/project-lifecycle/路径
      - 修正UPM的pointers字段
      - 移动错误位置的文档到正确路径

    情况B_信息过时:
      - 检查UPM的lastUpdateAt
      - 如果超过3天未更新，可能过时
      - 重新运行Step 7进度更新

    情况C_AI理解偏差:
      - 检查文档描述是否清晰
      - 增加上下文说明
      - 使用更明确的标记

  预防措施:
    - 使用标准路径，不擅自创建新路径
    - 及时更新文档，不延迟
    - 定期审核文档质量

问题3_文档偏差处理:
  症状:
    - 实际执行与计划严重偏差
    - 偏差未记录或记录不完整

  诊断方法:
    1. 对比计划 vs 实际
    2. 检查偏差记录章节
    3. 分析偏差原因

  解决方案:
    Step 1: 记录偏差
      - 使用标准偏差记录格式
      - 分析偏差原因
      - 评估偏差影响

    Step 2: 决策是否修正计划
      - 使用修正决策树
      - 选择适当修正策略

    Step 3: 执行修正
      - 追溯修正计划（如需）
      - 保留历史计划
      - 更新UPM的nextCycle

    Step 4: 验证修正效果
      - 在下一Cycle跟踪
      - 评估修正是否有效

  预防措施:
    - 提高工作量估算准确性
    - 增加技术预研
    - 使用偏差模式库优化估算

问题4_并发冲突处理:
  症状:
    - stateToken校验失败
    - 写入UPM时报告冲突
    - 多个AI同时修改同一文档

  诊断方法:
    1. 检查当前stateToken
    2. 检查最后更新时间
    3. 查看Git提交历史

  解决方案:
    策略A_重读-合并-重试:
      Step 1: 读取最新UPM状态
      Step 2: 合并自己的变更和最新状态
      Step 3: 重新计算stateToken
      Step 4: 再次尝试写入
      Step 5: 如再次冲突，最多重试3次

    策略B_创建UPMv2-MERGE段:
      IF 重试3次仍失败 THEN
        Step 1: 在UPM中创建临时MERGE段
        Step 2: 记录冲突上下文
        Step 3: 提供候选合并方案
        Step 4: 等待人工或AI合并

  预防措施:
    - 避免多个AI并发写入同一UPM
    - 使用任务队列串行化
    - 快速完成写入操作

问题5_UPM体量超限:
  症状:
    - UPM文档行数超过2500行
    - 读取速度明显变慢
    - 维护困难

  诊断方法:
    1. 统计UPM文档行数
    2. 分析哪些内容占据大量行数
    3. 确定可迁移的内容

  解决方案:
    Step 1: 识别可迁移内容
      - 历史Cycle详细记录（N-3之前）
      - 已完成任务的详细描述
      - 过期的风险记录

    Step 2: 迁移到周期文档
      - 移动详细内容到docs/project-lifecycle/week{N}/
      - 在UPM中保留索引条目

    Step 3: 更新历史索引表
      - 创建或更新"历史循环索引"章节
      - 提供指向周期文档的链接

    Step 4: 验证迁移效果
      - 确认UPM行数降低
      - 验证索引链接有效

  预防措施:
    - 定期检查UPM体量
    - 使用滚动窗口策略
    - 设置体量阈值告警
```

### 3.2 应急响应机制

```yaml
紧急状态定义:
  P0紧急状态（立即响应）:
    - 生产环境严重故障
    - 安全漏洞
    - 数据丢失风险
    - 构建完全失败

  P1紧急状态（2小时内响应）:
    - 质量严重回归（覆盖率<70%）
    - 关键功能阻塞
    - 里程碑严重延迟

  P2紧急状态（24小时内响应）:
    - 文档严重不一致
    - 多个任务延迟
    - 技术债务积累严重

应急响应流程:
  阶段1_快速评估（5-10分钟）:
    Step 1: 确认问题性质
      - 问题类型（构建/测试/性能/安全）
      - 影响范围（单模块/多模块/全系统）
      - 紧急等级（P0/P1/P2）

    Step 2: 初步诊断
      - 检查最近的变更（git log）
      - 查看技术验证记录
      - 分析错误日志

    Step 3: 决策响应策略
      IF P0紧急状态 THEN
        立即触发应急响应
      ELSE IF P1紧急状态 THEN
        2小时内启动处理
      ELSE IF P2紧急状态 THEN
        计划24小时内处理

  阶段2_问题定位（30分钟-2小时）:
    诊断方法:
      - 二分查找法定位引入问题的提交
      - 对比正常版本和问题版本
      - 分析依赖关系

    诊断工具:
      - git bisect
      - 差异比较工具
      - 日志分析工具

    输出:
      - 问题根本原因
      - 影响范围评估
      - 修复难度评估

  阶段3_快速修复（时间取决于复杂度）:
    修复策略:
      策略A_快速回滚:
        IF 问题是最近引入的 AND 无数据迁移 THEN
          git revert {problem-commit}
          重新运行技术验证
          部署回滚版本

      策略B_热修复:
        IF 问题明确且修复简单 THEN
          创建hotfix分支
          快速修复
          验证修复效果
          合并和部署

      策略C_临时绕过:
        IF 修复复杂但可绕过 THEN
          关闭问题功能
          启用备用方案
          计划彻底修复

  阶段4_验证恢复（30分钟）:
    验证项:
      □ 问题已解决
      □ 未引入新问题
      □ 质量指标恢复
      □ 功能正常

    全面验证:
      - 运行完整测试套件
      - 手动验证关键功能
      - 检查性能指标
      - 审查代码变更

  阶段5_总结复盘（1-2小时）:
    复盘内容:
      - 问题根本原因
      - 为什么未能预防
      - 修复过程是否高效
      - 如何避免重复

    输出文档:
      - 事故报告
      - 改进措施
      - 预防机制

业务连续性保障:
  降级策略:
    - 关键功能优先保证
    - 非核心功能可临时关闭
    - 用户体验优先于完整性

  回滚策略:
    - 保留最近3个稳定版本
    - 可快速回滚到任一版本
    - 回滚SOP文档

  备份恢复:
    - 定期备份关键数据
    - 验证备份可恢复
    - 恢复时间<1小时
```
