# 内容真实性规则

> **Version**: 2.0.0
> **Status**: Active
> **Purpose**: 确保所有文档和代码内容的真实性、准确性和一致性

---

## 1. 核心原则

### 1.1 三大支柱

| 支柱 | 说明 | 重要性 |
|------|------|--------|
| **真实性** | 所有信息必须基于事实 | 最高 |
| **准确性** | 数据和描述必须精确 | 最高 |
| **一致性** | 内容在各处保持统一 | 高 |

### 1.2 适用范围

- 文档内容 (Markdown, README, etc.)
- 代码注释
- 提交消息
- API文档
- 测试描述
- 配置文件注释

## 2. 禁止虚构信息

### 2.1 人员信息

```yaml
❌ 禁止:
  - 虚构人名: "张三", "John Doe", "测试用户"
  - 虚构邮箱: "test@example.com" (在正式文档中)
  - 虚构联系方式

✅ 正确做法:
  - 使用角色描述: "项目负责人", "前端开发者"
  - 使用占位符并标注: "[待填写]", "TBD"
  - 明确标记示例: "示例: user@example.com"
```

### 2.2 日期和时间

```yaml
❌ 禁止:
  - 使用未来日期作为已完成事项
  - 虚构历史日期
  - 不准确的时间戳

✅ 正确做法:
  - 使用实际日期: "2025-12-14"
  - 不确定时标注: "待定", "TBD"
  - 计划日期明确标记: "计划: 2025-12-20"
```

### 2.3 功能状态

```yaml
❌ 禁止:
  - 将未完成功能标记为已完成
  - 虚构测试覆盖率
  - 夸大功能描述

✅ 正确做法:
  - 准确标记状态: "进行中", "已完成", "计划中"
  - 使用实际数据: "覆盖率: 45%"
  - 明确说明限制: "当前仅支持..."
```

## 3. 日期规范

### 3.1 格式标准

| 场景 | 格式 | 示例 |
|------|------|------|
| 文档日期 | YYYY-MM-DD | 2025-12-14 |
| 时间戳 | ISO 8601 | 2025-12-14T14:30:00+08:00 |
| 年月 | YYYY-MM | 2025-12 |
| 相对日期 | 明确描述 | "3天前", "下周一" |

### 3.2 时区处理

```yaml
规则:
  - 默认使用本地时区 (UTC+8)
  - 跨时区场景标注时区
  - Git提交使用本地时间

示例:
  ✅ "2025-12-14T14:30:00+08:00"
  ✅ "2025-12-14 (北京时间)"
  ❌ "2025-12-14T14:30:00" (缺少时区)
```

### 3.3 文档日期更新

```yaml
更新时机:
  - 内容实质性变更时更新 updated_at
  - 格式调整不需更新
  - Changelog 必须记录重大变更

YAML Frontmatter 示例:
  ---
  created_at: '2025-12-01'
  updated_at: '2025-12-14'
  ---
```

## 4. 引用真实性

### 4.1 文件路径

```yaml
❌ 禁止:
  - 引用不存在的文件
  - 错误的相对路径
  - 过时的文件引用

✅ 正确做法:
  - 验证路径存在
  - 使用相对路径优先
  - 定期检查链接有效性
```

### 4.2 外部链接

```yaml
规则:
  - URL 必须可访问
  - 引用稳定的版本/永久链接
  - 标注访问日期 (对于可能变化的内容)

示例:
  ✅ [Conventional Commits](https://www.conventionalcommits.org/v1.0.0/)
  ✅ [Flutter Docs](https://docs.flutter.dev/) (2025-12)
  ❌ [某文档](http://example.com/temp-link)
```

### 4.3 代码引用

```yaml
规则:
  - 引用的代码必须存在
  - 行号引用要准确
  - 版本变更后及时更新

格式:
  - 文件:行号 → "src/auth.py:42"
  - 函数引用 → "AuthService.validateToken()"
```

## 5. 示例数据规范

### 5.1 标识规则

```yaml
必须标识:
  - 所有示例数据
  - 模拟数据
  - 测试数据

标识方式:
  - 注释: "// 示例数据"
  - 前缀: "example_", "mock_", "test_"
  - 文档说明: "> 以下为示例数据"
```

### 5.2 示例数据要求

```yaml
要求:
  - 格式正确 (符合实际数据格式)
  - 逻辑合理 (数值、关系合理)
  - 不包含敏感信息

示例:
  ✅ email: "user@example.com"  // 示例
  ✅ phone: "138****1234"       // 脱敏示例
  ❌ password: "123456"         // 真实弱密码
```

## 6. AI 生成内容规范

### 6.1 标识要求

```yaml
需要标识的场景:
  - AI 辅助生成的文档
  - AI 建议的代码
  - AI 翻译的内容

不需要标识:
  - AI 格式化/重排
  - AI 拼写检查
  - 常规代码补全
```

### 6.2 标识方式

```yaml
文档:
  > 本文档由 AI 辅助生成，已经人工审核

代码注释:
  # AI-assisted: 此函数逻辑由 AI 建议

提交消息:
  # 禁止添加 AI 签名 (见 git-commit.md)
```

### 6.3 审核要求

```yaml
AI 生成内容必须:
  - 人工验证准确性
  - 确认符合项目规范
  - 测试功能正确性 (代码)
```

## 7. 版本和变更追踪

### 7.1 版本号规范

```yaml
格式: MAJOR.MINOR.PATCH

变更类型:
  - MAJOR: 重大架构/内容变更
  - MINOR: 新增章节/功能
  - PATCH: 修正/优化
```

### 7.2 Changelog 规范

```yaml
必须记录:
  - 版本号
  - 变更日期
  - 变更内容摘要

示例:
  changelog:
    - version: 2.0.0
      date: '2025-12-14'
      changes: 重构文档结构，添加AI内容规范
```

## 8. 违规示例

### 8.1 严重违规

```yaml
案例 1 - 虚假完成状态:
  ❌ "功能已完成，测试通过" (实际未测试)
  ✅ "功能开发完成，待测试"

案例 2 - 虚构数据:
  ❌ "覆盖率达到 95%" (实际 45%)
  ✅ "当前覆盖率 45%，目标 80%"

案例 3 - 错误日期:
  ❌ created_at: '2025-01-01' (实际 12月创建)
  ✅ created_at: '2025-12-14'
```

### 8.2 轻微违规

```yaml
案例 1 - 示例未标识:
  ⚠️ const user = { name: "John" }
  ✅ const user = { name: "John" }  // 示例数据

案例 2 - 链接未验证:
  ⚠️ [详情](./old-doc.md)  // 文件已移动
  ✅ [详情](./new-location/doc.md)
```

## 9. 检查清单

### 9.1 文档创建时

- [ ] 日期使用实际日期
- [ ] 人员信息使用角色描述
- [ ] 功能状态准确
- [ ] 示例数据已标识
- [ ] 链接已验证

### 9.2 文档更新时

- [ ] updated_at 已更新
- [ ] Changelog 已记录
- [ ] 旧链接已修正
- [ ] 状态信息同步

### 9.3 代码提交时

- [ ] 注释内容准确
- [ ] 测试数据已标识
- [ ] 无敏感信息
- [ ] 版本号正确
