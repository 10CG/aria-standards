## 2. AI 记忆系统需求讨论框架

### 2.1 当前设计回顾

Mobile RPD 中定义的 AI 记忆系统：

```yaml
当前设计 (Mobile RPD):
  本地记忆:
    技术: SQLCipher 加密 SQLite
    功能:
      - 原始数据收集（操作、情境、内容）
      - 短期记忆缓存（30-90天）
      - 基础规则匹配
      - 简单模式识别
    特点: 离线可用、即时响应

  云端记忆 (Memos):
    技术: Memos 向量数据库 + 时序分析
    功能:
      - 深度语义分析（向量化）
      - 多跳时序搜索
      - 知识图谱构建
      - 长期记忆存储
    特点: 纯云端、需授权

  数据流:
    本地收集 → 批量上传 → Memos 分析 → 结果缓存
```

### 2.2 需要讨论的关键问题

#### Q1: 业务目标确认

**问题**: AI 记忆系统的核心业务价值是什么？

| 选项 | 描述 | 复杂度 |
|------|------|--------|
| A. 个性化推荐 | 基于历史行为推荐任务处理方式 | 中 |
| B. 行为模式分析 | 识别用户习惯，提供洞察报告 | 中 |
| C. 智能自动化 | 自动分类、标签、优先级建议 | 高 |
| D. 全量记忆检索 | "记住所有事"，支持自然语言查询 | 高 |
| E. 数字元神基础 | 为未来个性化 LLM 训练提供数据 | 极高 |

**待决定**: 优先实现哪些？MVP 范围是什么？

---

#### Q2: 技术方案选择

**问题**: 云端记忆服务使用什么技术？

| 选项 | 优点 | 缺点 | 复杂度 |
|------|------|------|--------|
| **A. Memos** (当前设计) | 现成方案、时序分析强 | 可能不够灵活、依赖第三方 | 中 |
| **B. 自建向量数据库** | 完全控制、可定制 | 开发成本高、需要专业知识 | 高 |
| **C. 云服务商方案** | 稳定、可扩展 | 成本、供应商锁定 | 中 |
| **D. 混合方案** | 灵活、可演进 | 架构复杂 | 高 |

**待决定**:
- Memos 是否仍是最佳选择？
- 是否需要评估其他方案？
- 自建的可行性和时间成本？

---

#### Q3: 本地与云端边界

**问题**: 本地记忆和云端记忆的职责边界如何划分？

```
当前设计:
┌─────────────────┐     ┌─────────────────┐
│   本地记忆       │     │   云端记忆       │
│ ───────────────  │     │ ───────────────  │
│ • 数据收集       │ ──→ │ • 深度分析       │
│ • 短期缓存       │     │ • 长期存储       │
│ • 基础规则       │ ←── │ • 结果缓存       │
│ • 离线可用       │     │ • 向量搜索       │
└─────────────────┘     └─────────────────┘
```

**讨论要点**:

| 问题 | 当前设计 | 待讨论 |
|------|----------|--------|
| 本地存储多久？ | 30-90天 | 是否足够？是否需要用户配置？ |
| 什么数据上传？ | 用户授权的数据 | 具体哪些字段？隐私等级如何？ |
| 离线能力范围？ | 基础规则匹配 | 是否需要更强的本地 AI？ |
| 云端依赖程度？ | 深度功能必须在线 | 是否接受？是否需要降级方案？ |

---

#### Q4: 隐私与合规

**问题**: 用户数据如何保护？

| 考量 | 当前设计 | 待确认 |
|------|----------|--------|
| 数据加密 | 本地 SQLCipher，传输 HTTPS | 云端静态加密？ |
| 用户授权 | 明确授权才上传 | 授权粒度？撤回机制？ |
| 数据删除 | 未明确 | 用户删除账号后数据如何处理？ |
| 合规要求 | 未明确 | GDPR？中国数据法规？ |

#### Q5: 版本演进路径

**问题**: AI 记忆系统如何分阶段交付？

```yaml
建议的演进路径:

Phase 4:
  目标: 数字元神

    - 个性化模型训练数据
```

**待决定**:
- Phase 1 的具体范围？

### D2: 云端记忆技术方案

**决策日期**: 2026-01-02

**放弃 Memos，采用组合架构**:

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     记忆协调层 (Memory Orchestration)                    │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │                    Mem0 Framework                                 │  │
│  │  支持 Episodic, Semantic, Procedural, Associative 四类记忆        │  │
│  └──────────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────────────────┐
│                      认知处理层 (Cognitive Processing)                   │
│  ┌─────────────────────────┐     ┌─────────────────────────┐           │
│  │    Zep/Graphiti         │     │     HippoRAG            │           │
│  │    时序知识图谱          │     │     多跳检索引擎         │           │
│  └─────────────────────────┘     └─────────────────────────┘           │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │              Triplex (3.8B) - 知识图谱提取                        │  │
│  └──────────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────────────────┐
│                        存储层 (Storage Layer)                           │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │                    PostgreSQL + pgvector                          │  │
│  │  Vectors (HNSW) | Graph (Edges) | Key-Value | Metadata (Time)     │  │
│  └──────────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘
```

**部署方式**:
- 本地 PVE 服务器部署（自托管）
- 大模型调用外部 API
- 多用户共享（用户数量不多）

### D3: 本地/云端边界定义

**决策日期**: 2026-01-02

**本地存储**: 7-30天可配置

**离线操作 (以记录为主)**:
- 任务 CRUD 操作
- 四象分类（规则匹配）
- 本地操作记忆收集
- 偏好学习数据存储
- 基础对话（本地 LLM，需评估硬件）
- 任务状态查询
- 提醒设置

**在线操作 (需要 PVE/云端)**:
- 四相灵宠内部讨论
- 深度分析与优化建议
- 交互式问卷
- 方案生成
- 预测分析
- 数据同步
- 记忆上传与云端分析

**移动端本地记忆**:
- 保持 SQLite + SQLCipher
- 需评估硬件支持能力（Triplex 3.8B 可能不适合移动端）
- 复杂处理上传到 PVE 服务器

### D4: 隐私合规要求

**决策日期**: 2026-01-02

| 要求 | 决策 |
|------|------|
| 数据托管 | 完全自托管（PVE 服务器） |
| 数据加密 | 本地 SQLCipher + 传输 HTTPS + 云端静态加密 |
| LLM API 调用 | 脱敏加密后发送，避免数据泄露 |
| 数据脱敏 | 存储时脱敏 + API 调用时脱敏 |

## 6. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 讨论周期过长 | 延迟开发 | 设定讨论截止日期，未决项记录后继续 |
| 需求范围蔓延 | 复杂度失控 | 严格 MVP 范围，Phase 1 最小化 |
| 技术方案变更 | 已有设计作废 | Mobile 端保持抽象接口，减少硬编码 |
| 移动端硬件限制 | Triplex 无法运行 | 复杂处理上传 PVE，移动端保持轻量 |
| 多技术栈集成 | 架构复杂度高 | 分阶段集成，Phase 1 只部署基础设施 |
