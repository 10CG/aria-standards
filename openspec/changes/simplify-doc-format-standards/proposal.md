# Unify Documentation Format Standards

> **Level**: Full (Level 3 Spec)
> **Status**: Draft
> **Created**: 2025-12-29

## Why

1. **规范分散**: 文档格式规范分散在 `docs/project-standards/` 和 `standards/` 两处
2. **缺乏体系**: 没有完整梳理项目开发周期所需的全部文档类型
3. **查漏补缺**: 部分文档类型可能缺少格式规范，部分规范可能过度设计

需要先系统梳理"项目需要哪些文档类型"，再统一其格式规范到 `standards/` 目录。

## What

建立完整的文档格式规范体系：
1. **梳理文档类型** - 基于项目开发周期，识别所有需要的文档类型
2. **查漏补缺** - 检查每种文档类型是否有对应的格式规范
3. **统一规范** - 将通用的格式规范整合到 `standards/` 目录
4. **简化设计** - 移除过度设计，让 Skills 承担动态验证

### Key Deliverables

- **文档类型清单**: 完整的项目文档类型分类和用途说明
- **格式规范矩阵**: 每种文档类型对应的格式规范（现有/缺失/需简化）
- **统一的 standards/ 规范**: 通用文档格式规范
- **Skills 职责映射**: 哪些格式由 Skills 动态验证

## Project Development Lifecycle & Document Types

**核心问题**: 当前 AI-DDD 十步循环主要覆盖"开发实现"阶段，大型项目需要的需求/设计/运维文档缺少对应规范。

### 项目规模定位

**本项目规模**: 1-3 人小团队

**适配原则**:
- 文档即代码 - 能用代码/测试表达的，不写文档
- 轻量优先 - 用 OpenSpec 替代正式 PRD/设计文档
- AI 友好 - 文档服务于 AI 辅助开发

### 小团队文档类型清单 (最终版)

#### 必需 (9 种)

| 文档类型 | 用途 | 格式规范 | 说明 |
|----------|------|----------|------|
| **CLAUDE.md** | AI 上下文入口 | ✅ 有 | - |
| **README.md** | 项目/模块说明 | ⚠️ 需模板 | - |
| **PRD (轻量版)** | 产品/功能需求概述 | ❌ 需创建 | 宏观需求描述 |
| **User Story** | 细粒度用户需求 | ❌ 需创建 | PRD 拆解后的需求点 |
| **OpenSpec Proposal** | 技术实现方案 | ✅ 有 | 基于需求设计 |
| **OpenSpec Tasks** | 任务分解 | ✅ 有 | - |
| **ARCHITECTURE.md** | 代码架构说明 | ✅ L0/L1/L2 | 替代 HLD/LLD |
| **API Contract** | 接口定义 | ✅ OpenAPI | - |
| **UPM** | 进度追踪 | ✅ 有 | - |

#### 推荐 (4 种)

| 文档类型 | 用途 | 格式规范 | 说明 |
|----------|------|----------|------|
| **OpenSpec Design** | 复杂决策记录 | ⚠️ 需模板 | 复杂功能才需要 |
| **Changelog** | 版本变更记录 | ⚠️ 需模板 | Keep a Changelog |
| **Summaries** | 摘要文件 | ✅ 有 | Token 优化 |
| **Standards** | 开发规范 | ✅ 有 | 团队约定 |

#### 可选 (按需)

| 文档类型 | 何时需要 |
|----------|----------|
| Release Notes | 有外部用户时 |
| Deployment Guide | 部署复杂时 |
| Test Strategy | 测试复杂时 |

#### 不需要 (小团队可省略)

| 文档类型 | 原因 |
|----------|------|
| BRD (商业需求) | → 小团队不需要独立商业文档 |
| SRS (需求规格) | → 轻量 PRD + User Story 替代 |
| HLD/LLD | → ARCHITECTURE.md 替代 |
| 正式测试计划/用例 | → 代码测试 + test-verifier 替代 |

| 项目计划/风险管理 | → UPM + Tasks 替代 |
| 会议纪要 | → 小团队口头沟通 |
| Bug Report 模板 | → GitHub Issues 足够 |

### PRD → User Story → OpenSpec 三层关系

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         需求到实现的文档流                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  PRD (产品需求)              User Stories            OpenSpec Proposals │
│  ┌─────────────────┐        ┌─────────────┐        ┌─────────────────┐ │
│  │ 背景与目标       │        │ Story 1     │        │ Proposal A      │ │
│  │ 用户角色        │   →    │ Story 2     │   →    │ Proposal B      │ │
│  │ 功能范围        │        │ Story 3     │        │ Proposal C      │ │
│  │ 非功能需求      │        │ ...         │        │ ...             │ │
│  │ 成功指标        │        │             │        │                 │ │
│  └─────────────────┘        └─────────────┘        └─────────────────┘ │
│                                                                         │
│  宏观: What & Why            细粒度: 用户价值         技术: How          │
│  (一个功能/版本一个)          (多个需求点)            (每个实现一个)       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**层级说明**:

| 层级 | 文档 | 粒度 | 关注点 | 数量关系 |
|------|------|------|--------|----------|
| L1 | **PRD** | 功能/版本 | What to build & Why | 1 PRD |
| L2 | **User Story** | 需求点 | 用户价值 | → N Stories |
| L3 | **OpenSpec Proposal** | 实现方案 | How to build | → M Proposals |

**使用场景**:
- **大功能/新版本**: PRD → User Stories → OpenSpec Proposals (完整流程)
- **中等功能**: User Stories → OpenSpec Proposals (跳过 PRD)
- **小修复**: 直接 OpenSpec Proposal (隐含需求)

### 过度设计案例

1. **元数据格式 v6**: 要求 ISO 8601 精确到秒、多层嵌套 YAML
2. **四段式命名强制**: `[领域]-[类型]-[主题]-[子主题].md` 难以执行
3. **组件引用系统**: `<!-- BEGIN:COMPONENT_REFERENCE -->` 从未实际使用
4. **生成脚本**: 大量设计但不存在的自动化脚本

## Impact

| Type | Description |
|------|-------------|
| **Positive** | 减少文档维护负担，提高 AI 读取效率，降低新人学习曲线 |
| **Positive** | Token 消耗降低（移除冗余规范） |
| **Risk** | 清理过程可能误删有用内容 → 先归档再删除 |
| **Risk** | Skills 职责边界不清 → 创建职责映射表 |

## Design Principles

### 保留原则

1. **L0/L1/L2 架构文档分层** - 已验证有效
2. **OpenSpec proposal/tasks 模板** - 简洁实用
3. **Summary 摘要文件** - Token 优化核心

### 简化原则

1. **元数据只保留必要字段**: Version, Status, Updated
2. **命名规范软性指导**: 推荐而非强制
3. **Skills 承担验证**: 格式检查由 skill 执行，不嵌入文档

### Skills 职责映射

| 格式类型 | 负责 Skill | 说明 |
|----------|-----------|------|
| Git Commit | `commit-msg-generator` | 生成符合规范的提交消息 |
| OpenSpec 文档 | `spec-drafter` | 生成规范的 proposal |
| 架构文档 | `arch-update` | 更新时验证格式 |
| UPM 进度 | `progress-updater` | 维护 UPM 格式 |
