# Clarify Phase A Task Transformation Pipeline

> **Level**: Full (Level 3 Spec)
> **Status**: Draft
> **Created**: 2025-12-19
> **Module**: standards

## Why

Current Phase A documentation (A.0-A.3) lacks clear boundaries between A.1 (Spec Management), A.2 (Task Planning), and A.3 (Agent Assignment), leading to:

1. **Confusion about task granularity**: When does a "task" from A.1's tasks.md become a "TASK-{NNN}" in A.2?
2. **Unclear document roles**: Should tasks.md be updated after A.2 refinement?
3. **Overlapping responsibilities**: Is A.3 (Agent Assignment) substantial enough to be a separate step, or should it merge with A.2?

Without clear boundaries, AI assistants and developers struggle to understand:
- Which document is the "single source of truth" for execution
- How to maintain consistency between OpenSpec's tasks.md and AI-DDD's detailed task specifications
- When and how to synchronize these two task views

## What Changes

### 1. Formalize Two-Layer Task View Architecture

Establish a dual-document system with clear roles:

| Document | Layer | Source | Format | Purpose | Update Mechanism |
|----------|-------|--------|--------|---------|------------------|
| **tasks.md** | Coarse-grained | A.1 | OpenSpec standard | High-level progress tracking, OpenSpec compatibility | Checked off during B.2 execution |
| **detailed-tasks.yaml** | Fine-grained | A.2 | Structured YAML | AI execution basis, scheduling foundation | Generated by A.2, enhanced by A.3 |

### 2. Define Clear Step Boundaries

**A.1 (Spec Management) - OpenSpec Alignment Layer**:
- **Responsibility**: Create OpenSpec-compliant Spec and coarse-grained checklist
- **Output**: `proposal.md` + `tasks.md` (OpenSpec standard)
- **Unique Value**:
  - OpenSpec ecosystem compatibility
  - High-level functional view
  - Non-technical stakeholder comprehension

**A.2 (Task Planning) - Execution Preparation Layer**:
- **Responsibility**: Transform functional checklist into executable atomic task specifications
- **Output**: `detailed-tasks.yaml` (TASK-{NNN} with metadata)
- **Unique Value**:
  - Precise time estimates and dependency management
  - File-level deliverable specifications
  - Parallel scheduling and risk assessment support

**A.3 (Agent Assignment) - Resource Matching Layer**:
- **Responsibility**: Match tasks with specialized agents, set verification criteria
- **Output**: Enhanced `detailed-tasks.yaml` (with agent + verification)
- **Unique Value**:
  - Domain expertise-based optimal matching
  - Task-specific verification standards
  - Execution environment and quality gate preparation

### 3. Establish Task Transformation Pipeline

Define explicit transformation stages:

```
Stage 1 (A.1 Output) - Coarse-grained Functional Checklist
─────────────────────────────────────────────────────────
File: changes/{feature}/tasks.md
Format: OpenSpec checkbox markdown

Example:
  ## 1. Database Setup
  - [ ] 1.1 Add OTP secret column to users table
  - [ ] 1.2 Create OTP verification logs table

  ## 2. Backend Implementation
  - [ ] 2.1 Add OTP generation endpoint
  - [ ] 2.2 Modify login flow to require OTP

Characteristics:
  - Functional checklist format
  - High-level deliverables
  - No dependencies specified
  - No time estimates
  - Aligned with OpenSpec scaffolding


Stage 2 (A.2 Output) - Refined Executable Tasks
──────────────────────────────────────────────
File: changes/{feature}/detailed-tasks.yaml
Format: Structured YAML

Example:
  tasks:
    - id: TASK-001
      parent: "1.1"  # Links to tasks.md section 1.1
      title: Add OTP secret column to users table
      complexity: M (Medium, 2-4h)
      dependencies: []
      deliverables:
        - backend/migrations/add_otp_secret_column.sql
        - backend/src/models/user.py (update User model)

    - id: TASK-002
      parent: "2.1"
      title: Create OTP generation endpoint
      complexity: L (Large, 4-8h)
      dependencies: [TASK-001]
      deliverables:
        - backend/src/routes/otp_routes.py
        - backend/src/services/otp_service.py

Characteristics:
  - TASK-{NNN} unique identifiers
  - Complexity estimates with hour ranges
  - Explicit dependency chains
  - File-level deliverable specifications
  - 4-8 hour atomic granularity
  - Parent links to tasks.md for traceability


Stage 3 (A.3 Output) - Agent-Assigned Tasks
──────────────────────────────────────────
File: changes/{feature}/detailed-tasks.yaml (enhanced)
Format: Structured YAML with agent metadata

Example:
  tasks:
    - id: TASK-001
      parent: "1.1"
      title: Add OTP secret column to users table
      complexity: M (Medium, 2-4h)
      dependencies: []
      deliverables:
        - backend/migrations/add_otp_secret_column.sql
        - backend/src/models/user.py
      agent: backend-architect
      reason: Database schema design expertise, migration management
      verification:
        - Migration runs successfully
        - Column exists in schema with correct type
        - User model updated and tests pass

Characteristics:
  - Inherits all A.2 task metadata
  - Specialized agent assignment
  - Assignment rationale based on expertise
  - Task-specific verification criteria
  - Ready for Phase B execution
```

### 4. Synchronization Mechanism

**Forward Sync (A.1 → A.2 → A.3)**:
- A.2's `task-planner` skill reads tasks.md and generates detailed-tasks.yaml
- A.3 enhances detailed-tasks.yaml with agent assignments
- Parent field in detailed-tasks.yaml maintains traceability

**Backward Sync (B.2 → tasks.md)**:
- When TASK-{NNN} is completed in Phase B, corresponding tasks.md item is checked off
- Skills (e.g., task-progress-updater) handle this synchronization
- OpenSpec CLI (`openspec list`) can show progress from tasks.md

**Conflict Resolution**:
- tasks.md is the "progress view" (checkboxes)
- detailed-tasks.yaml is the "execution spec" (single source of truth for implementation)
- If discrepancies arise, detailed-tasks.yaml takes precedence for execution decisions

## Impact

### Positive

1. **Clear Separation of Concerns**:
   - Each step (A.1, A.2, A.3) has a well-defined, non-overlapping responsibility
   - Reduces confusion for both AI assistants and human developers

2. **OpenSpec Compatibility**:
   - Maintains tasks.md in standard OpenSpec format
   - Compatible with `openspec` CLI tools (list, validate, show)

3. **Enhanced Execution Precision**:
   - detailed-tasks.yaml provides all metadata needed for scheduling, risk assessment, and quality gates
   - File-level deliverables enable precise progress tracking

4. **Improved Traceability**:
   - Parent field links detailed tasks back to high-level checklist
   - Easier to understand "why" behind atomic task breakdown

5. **Skill-Driven Automation**:
   - Clear boundaries enable skills (task-planner, etc.) to automate transformations
   - Reduces manual synchronization burden

### Risks and Mitigation

| Risk | Impact | Mitigation |
|------|--------|------------|
| **Document Divergence** | tasks.md and detailed-tasks.yaml get out of sync | 1. Skills handle synchronization automatically<br>2. Validation checks detect discrepancies<br>3. Clear documentation of which is "source of truth" for what |
| **Increased Complexity** | Two documents to maintain instead of one | 1. Each serves a distinct purpose<br>2. Automation via skills reduces manual effort<br>3. Benefits (clarity, compatibility) outweigh cost |
| **Learning Curve** | Developers must understand two-layer model | 1. Clear documentation with examples<br>2. Visual diagrams in Phase A docs<br>3. Skills abstract complexity for most users |
